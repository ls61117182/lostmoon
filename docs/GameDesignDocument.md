# 《孤胆谢尔曼》手机游戏策划案

> 基于同名桌游规则改编的单人战棋手游  
> 引擎：Cocos Creator 3.8.8  
> 平台：iOS / Android（横屏为主，竖屏适配）  
> 类型：单机回合制 / 战棋 / Roguelite  
> 版本：v0.1（首版立项稿）

---

## 一、项目概述

### 1.1 游戏名称
- 中文：《孤胆谢尔曼》
- 英文（暂定）：**Lone Sherman: Tactical Tank Commander**

### 1.2 游戏背景
玩家在二战法国战场指挥一辆美军 M4 谢尔曼坦克及其乘员，独自面对由 AI 驱动的德军坦克与步兵，完成 12 个不同主题的战术任务。游戏不追求绝对历史精确，强调"剧情驱动 + 战术决策 + 骰运紧张感"的体验。

### 1.3 核心卖点
1. **真正的单人战棋**：所有敌方行动由确定性规则表（AI 表）驱动，不需要联网、不需要对手。
2. **掷骰即战场**：以 5–7 颗骰子（虚拟）作为唯一行动资源，每一次摇骰都是一次决策。
3. **乘员 RPG 化**：5 名乘员（车长、炮手、装填手、驾驶员、副驾驶）各自影响掷骰数量，可阵亡、可补员，单局养成。
4. **纸面规则数字化**：把桌游中"找表 → 修正 → 比较"的繁琐流程自动化，留下决策乐趣。
5. **12 关任务 + 战役模式**：可单关游玩，也可串联成战役（保留炮塔受损、痛痪、着火程度跨关延续）。

### 1.4 目标用户
- 战棋 / 桌游爱好者（Memoir'44、Combat Commander、Patton's Best 玩家）
- 喜欢硬核单机战术、Roguelite 关卡的轻度核心玩家
- 二战题材爱好者

### 1.5 商业模式
- 买断制（建议 ¥18–¥30）+ 任务包 DLC（后续追加 6–12 关）
- 不含内购、不含广告，主打"纯净战棋"体验

## 二、核心玩法循环

### 2.1 单关流程

```
进入任务 → 设置阶段（自动放置敌方）
   ↓
进入回合循环（7 阶段）：
  ① 移除谢尔曼烟雾
  ② 决定车长舱盖位置（开/关）
  ③ 谢尔曼行动（移动 / 攻击 / 杂项）—— 玩家核心决策
  ④ 移除德军烟雾
  ⑤ 着火程度检定
  ⑥ 德军坦克 AI 行动
  ⑦ 回合结束事件（援军、地雷、断图卡等）
   ↓
判定胜负（达成任务目标 / 谢尔曼被摧毁 / 全员阵亡）
   ↓
战后结算（乘员补充、状态延续）→ 下一任务 / 返回主菜单
```

### 2.2 战役流程
- 12 关任务可顺序通关，也可随机抽选构筑"战役"。
- 关与关之间允许"恢复"：补充 1 名阵亡乘员（玩家选择）。
- **跨关延续**：炮塔受损、痛痪状态、着火程度、装填状态全部保留，制造决策连锁。

### 2.3 一次"行动阶段"（玩家核心决策）拆解
```
① 由**当前子阶段（移动 / 攻击 / 杂项）**、**谢尔曼当前格地形**与**乘员存活 / 舱盖**等，按 §3.6.1 表格规则自动计算本阶段可掷骰子数（再经配置的上下限钳制）
② 玩家点击"摇骰"按钮，按该数量播放骰子动画落定
③ 玩家把骰子按顺序（A 移动 / B 攻击 / C 杂项）拖入行动槽
   - A 与 B 顺序可互换（玩家选择"先移动后攻击"或"先攻击后移动"）
   - C 必须最后执行
④ 每个行动按对应规则结算（移动动画、命中判定、装填、烟雾等）
⑤ 阶段结束，进入下一阶段
```

> **设计要点**：要在手机端把"骰子拖动 + 行动槽匹配"做得直观，是 UX 最大挑战之一。详见第六章。

**桥梁（叠加在水域格上，非独立地形类型）**

- 桥梁**不是**与上表并列的一种地形，仅能**叠加**在 `t === "w"` 的水域格上。配置桥梁的水域格相当于"在水面上架了一座桥"，**坦克 / 卡车可进入**；不带桥梁的水域格仍然完全不可入。
- **骰子规则与公路相同**：桥梁格的移动 / 攻击 / 杂项基础骰数全部按公路读取，移动力消耗也按公路计（与基底"水域"完全脱钩，避免出现"+0 颗骰、不可入但又能动"的死锁）。
- **「按公路触发」的规则一律视桥梁为公路**：除骰子 / 移动力外，所有以「站在公路格」为前置条件的事件——典型如 §3.8 / 任务事件表中的 **`road_mine` 公路地雷**——也同样在桥梁格上触发；实现层统一通过 `effectiveDiceTerrain(tile)` 把"水域+桥梁"折算为 `'road'` 后再判定，避免出现"桥上吃骰加成、却躲掉地雷"的不一致。
- **必须配置两端方向**（关卡 JSON：`TileDef.br = [a, b]`，`a, b ∈ 0..5` 且不重复）。这两个方向定义了桥梁向**外**贯通的两条边；其余 4 条边视为水面，车辆不可越。
- **越境规则**：车辆（坦克 / 卡车）只能从「桥梁两端方向之一」**驶入**或**驶出**桥梁格——
  - 进入：从相邻格 A → 桥梁格 B 时，必须满足 `dir(B → A) ∈ B.br`（即 A 落在 B 的某一桥梁端方向上）；
  - 离开：从桥梁格 B → 相邻格 C 时，必须满足 `dir(B → C) ∈ B.br`（即 C 落在 B 的某一桥梁端方向上）。
  - 两个相邻桥梁格相连时，"出"与"入"两条约束同时生效——两端必须**同时对齐**才允许跨越（即两座桥的桥端方向都覆盖那条物理边）。
- 视线 / 树篱：桥梁不影响视线（与水域一致：不阻挡），桥梁本身不附带树篱修正。
- 撤离 / 驶出：若桥梁格的某一端方向指向地图外，车辆可从该端**驶离地图**（与公路出口相同），此约束同样适用于谢尔曼撤离 (`destroy_kind_evac`) 和卡车末格 `truckPath.exitDir`：方向必须落在 `br` 端之一，否则会被边向校验拒绝。

**建筑（叠加在基底格上，非独立地形类型）**

- 建筑**不是**与上表并列的一种地形，而是在**公路 / 田地 / 泥地**等基底格上叠加的标记（美术可用房屋图案表示）。
- 基底为**田地**或**泥地**且带建筑时，在规则与任务表述中一并视为**农场**；基底为**公路**且带建筑时，视为**村庄**。
- 进入与掩护：**坦克可以进入**带建筑的格子；对**被攻击目标**而言，目标格有建筑仍提供掩护（§3.4 Step 1「目标位于建筑格 +1」）。
- 视线：从建筑**以外**的格子发出的射线，若路径上存在**中间格**（非起点、非终点）为带建筑格，则**被遮挡**；建筑格作为视线的**起点**或**终点**时，该格上的建筑**不**截断该条视线（建筑内单位可由此格向外开火）。林地仍阻挡视线；**水域不再阻挡视线**（仅保留「任何单位不可入」的进入限制）；树篱不减损视线、仅按 §3.4 修正命中（见上表）。

### 3.3 朝向、视线与射程
- 朝向 = 坦克车体（**不是炮塔**）面对的 6 个格边方向之一。
- 视线沿 6 个直线方向无限延伸（射程不限）：**穿过**路径上的林地，或**作为中间格**（非起止格）的带建筑格时，视线被阻挡；**水域不阻挡视线**；树篱不截断视线，但按 §3.4 修正命中。建筑格作为视线**起点**（例如建筑格内的坦克向外射击）或**终点**时，该格不视为沿该方向的视线阻挡物。
- 攻击方向决定使用哪一面装甲值（前 / 前侧 / 后侧 / 后），是核心战术深度。
- **方向圈**：以坦克为中心的 6 格分为前 / 前侧（×2）/ 后侧（×2）/ 后。
- **装甲面判定（实现约定）**：取从**目标格心**指向**攻击者格心**的六向射线（与主炮视线「攻击者与目标共线」一致；代码里与 `directionTo(目标, 攻击者)` 对齐），再相对车体 `facing` 求差得到前 / 前侧 / 后侧 / 后。若任务 JSON 里 `facing` 与美术车体「前」不一致，会出现「体感前后反了」——应改关卡朝向而非改掷骰公式。

### 3.4 战斗系统（攻击 = 三步走）

#### Step 1：能否命中？（掷 2d6）
计算"命中所需点数"：

```
命中所需 = 体型 + 距离（六格距离）
         + Σ 树篱数 (每穿过 1 个 +1；紧挨攻击者的树篱不计)
         + 目标位于建筑格 (+1)
         + 目标有烟雾掩护 (+1)
         + 目标处于隐蔽状态 (+2)
         + 目标在攻击方背后 (+1)
```

掷 2d6，**两骰之和 > 命中所需** → 命中。  
**额外规则**：掷出 2 个相同点数的命中（"对子"）时，若车长处于"打开舱盖"位置 → 车长阵亡（即便弹开也致命）。

> **"紧挨攻击者的树篱不计"细则**（实现：`HexMap.countHedgesAlong`）：
>
> 路径上的某段树篱若**与攻击者紧挨**，则不计入命中所需。"紧挨攻击者"指：
> 1. 该树篱与攻击者**处于同一格子**（即攻击者格 `hedges[dir(攻击者→第一邻格)]` 命中）；
> 2. 或该树篱所处格子**与攻击者相邻**且**树篱方向为攻击者所在格**（即第一邻格 `hedges[dir(第一邻格→攻击者)]` 命中）。
>
> 这两种写法对应**同一条物理边**（攻击者射出第一步穿过的那一段），实现上从路径第二条边起累计、并对 `i ≥ 1` 的每条边检查双侧编码（`a.hedges[dirAB] || b.hedges[dirBA]`），命中计 1 不重复累加。
> 直觉解释：自家篱笆一侧是熟悉地形，开火时不会被自家树篱遮挡视线，因此不增加难度。

#### Step 2：造成伤害？（掷 1d6）
```
d6 ≥ 目标对应面装甲 - 攻击方穿甲值  → 造成伤害
```
对应面取决于攻击方向（前/前侧/后侧/后）。

#### Step 3：怎样的伤害？（掷 1d6 查表）

| d6 | 谢尔曼 受伤 | 德军坦克 受伤 |
|---|---|---|
| 1 | 被摧毁 | 受损 → 被摧毁 |
| 2 | 阵亡检定（见乘员表） | 受损 → 被摧毁 |
| 3–4 | 着火 / 着火程度 +1 | 受损 → 被摧毁 |
| 5 | 炮塔受损 | 被摧毁 |
| 6 | 痛痪 | 被摧毁 |

> 谢尔曼"阵亡检定"：再掷 1d6（仅 1 次）映射到乘员 1–5；若掷出 6 且车长处于打开舱盖 → 车长阵亡。

### 3.5 着火 / 修复 / 状态系统

谢尔曼与敌方坦克在战斗中会累积可被再次解除的持久状态。下文按"总表 → 视觉 → 回合内检定 → 解除逻辑 → MVP 占位"五步展开，便于策划 / 程序 / 美术三方对齐。

#### 3.5.1 状态总表

| 状态 | 代码字段 | 触发 | 持续效果 | 解除方式 |
|------|---------|------|----------|----------|
| 着火程度 | `fireLevel: number`（0..∞，仅谢尔曼） | 谢尔曼受"着火"伤害（§3.4 Step 3，d6 = 3 或 4），每次 +1 | 阶段⑤"着火程度检定"：掷 N = fireLevel 颗 d6，每颗按 §3.4 Step 3 结果再走一轮伤害；数值也是"火苗烈度"的视觉强度依据 | 灭火行动（杂项 6 点）-1，最低 0；`fireLevel` 归 0 时橙色外观立即回归阵营色 |
| 受损 / 起火（敌） | `damaged: boolean` | 德军坦克任一次有效击穿（§3.4 d6 = 1..4 均触发）；谢尔曼任何非摧毁击穿也会置 true | **德军**：下次击穿直接升级为"摧毁"（MVP 语义）；**谢尔曼**：仅作为"曾受伤"的累积标记，具体惩罚由下面 3 行负责 | 德军不可解除；谢尔曼 `fireLevel=0` 且其他持续状态解除后视觉回归，但字段本身保留（用于"曾受伤"统计） |
| 炮塔受损 | `turretDamaged: boolean` | 伤害结果 = "炮塔受损"（§3.4 d6 = 5，仅谢尔曼条目 ） | 不能使用主炮射击 —— 攻击阶段 5 / 6 点（主炮骰）与杂项 1 点"炮手主炮"选项被禁用；右侧状态面板"炮塔"行显示 **受损** | 修复行动（杂项 4 点 或 5 点"烟/修"选"修"） |
| 瘫痪 | `paralyzed: boolean` | 伤害结果 = "瘫痪"（§3.4 d6 = 6，仅谢尔曼条目 ） | 不能前进 / 转向 / 后退 —— 移动阶段所有骰面动作一律失败；状态面板"机动"行显示 **瘫痪**；若后续被推入泥地则完全无法行动（§3.2 规则特例） | 修复行动（杂项 4 点 或 5 点二选一） |
| 隐蔽 | `hidden: boolean` | **谢尔曼**：杂项阶段对子（两颗同点）执行"隐蔽"动作；**德军坦克**：AI 表 `conceal` 动作（见 §3.7） | 被攻击命中所需 +2（§3.4 Step 1 修正项） | **任何坦克（含谢尔曼 / 德军坦克）执行 前进 / 转向 / 后退 任意一次移动动作后立即移除**；`BattleScene.breakConcealment(unit)` 在 `tryDriveSherman` / `tryTurnSherman` / 对子驾驶员 & 副驾驶 / 撤离驾驶动作以及敌方 `executeEnemyAction` 的 `turn` / `advance` / `reverse` 分支统一调用 |
| 烟雾掩护 | `smoked: boolean` | 谢尔曼：杂项 5 点"烟/修"选"烟"；德军：AI 表 `smoke` 动作 | 被攻击命中所需 +1（§3.4 Step 1 修正项）；可叠加建筑掩护 / 隐蔽 | 阶段①（谢尔曼）或阶段④（德军）开始时自动清除，**仅持续 1 回合** |
| 主炮已装填 | `loaded: boolean` | 关卡初始默认 `true`；攻击阶段 1 / 2 点"装填"或杂项 1 点"装填"命中 | 可开主炮（攻击 5 / 6 点或杂项 1 点"主炮射击"）；开火后自动清空 | 开火消费；炮塔受损不影响装填（可装填但不能开火） |
| 舱盖打开 | `hatchOpen: boolean` | 玩家在阶段② 手动切换（见 §2.1） | **移动 / 攻击子阶段**骰数 +1（§3.6.1）；杂项子阶段不加舱盖骰、另有车长存活 +1（§3.6.1）。§3.4 Step 1 掷出"对子命中"且舱盖开 → 车长阵亡风险；狙击手 / 斯图卡等事件也以此为前提 | 玩家手动关闭；进入本回合任一子阶段（A / B / C）后锁定到本回合结束 |
| 摧毁 | `destroyed: boolean` | §3.4 Step 3 d6 = 1 / 2（谢尔曼）或 1 / 2 / 3 / 4 再次击穿（德军）；阶段⑤着火检定也可能触发 | 单位从游戏退出 —— 不再行动、不阻塞移动（视作残骸，可被跨过） | 不可逆 |

> **叠加规则**：`fireLevel > 0` / `turretDamaged` / `paralyzed` / `hidden` / `smoked` 完全正交，可同时存在；任一解除不影响其他。
>
> **字段重载说明**：GDD 术语的 "受损（德军）" 与 "起火（谢尔曼）" 在代码里共用 `damaged: boolean`，仅外显文本不同；谢尔曼的"真起火强度"以 `fireLevel` 为准。

#### 3.5.3 回合流程中的状态检定

> 以下顺序对应 §2.1 单关回合循环 7 阶段，只列出与本节状态有关的步骤：

- **阶段① 移除谢尔曼烟雾**：若 `sherman.smoked=true`，开局置 false。
- **阶段② 决定舱盖位置**：玩家点"舱盖"行切换；进入后续子阶段后锁定。
- **阶段③ 谢尔曼行动**：
  - 移动 / 攻击 / 杂项 子阶段内凡受状态约束的动作（炮塔受损→禁主炮、瘫痪→禁移动…）在骰子托盘对应面直接表现为"无可用动作"。
  - 隐蔽与前进 / 转向 / 后退互斥：一旦消耗移动动作立即清除 `hidden`（同条规则在阶段⑥德军坦克 AI 转向 / 前进 / 后退动作上同样生效；见 §3.5.1 解除方式列）。
- **阶段④ 移除德军烟雾**：遍历所有存活敌方，`smoked` 统一置 false。
- **阶段⑤ 着火程度检定**：若 `sherman.fireLevel > 0`，掷 N 颗 d6，每颗按 §3.4 Step 3 谢尔曼列再走一次伤害结算（可能连锁触发"摧毁 / 瘫痪 / 炮塔受损 / 阵亡检定 / 起火 +1 / 受损"）。**该阶段是关卡核心压力来源之一，迫使玩家在 A / B / C 骰分配上为"灭火"骰留额度。**
- **阶段⑥ 德军坦克 AI 行动**：AI 表 `smoke` / `repair` 动作同样消费骰并修改对应状态。
- **阶段⑦ 回合结束事件**：结果亦可能触发"着火"、"瘫痪"等（见 §3.8）。

#### 3.5.4 解除方式 / 修复优先级

| 解除动作 | 骰面来源 | 实装 UI 行为 |
|---------|---------|-------------|
| 灭火（杂项 6 点） | `classifyMiscDie(6)='fire_suppress'` | 点骰 → 若 `fireLevel > 0` 直接 `-1` 并浮字"起火程度 -1"；否则浮字"无起火"并弃骰 |
| 修复炮塔 / 瘫痪（杂项 4 点） | `classifyMiscDie(4)='repair'` | 若两者并存 → 弹 2 选 1 菜单；若只有一项 → 直接执行并浮字"XX 已修复"；都无 → 浮字"无需修复"并弃骰 |
| 修复（杂项 5 点 "烟/修" 选 "修"） | `classifyMiscDie(5)='smoke_or_repair'` | 同上，并额外在菜单首行显示"施放烟雾"（MVP 占位） |
| 隐蔽（杂项 对子 `concealment`） | 任意 misc 阶段存在两颗同点未用骰 | 点其中一颗 → 菜单追加"进入隐蔽（+同点骰）"；选中即消耗两颗同点骰并置 `sherman.hidden=true`；下次移动动作自动清除 |
| 烟雾（杂项 5 点） | `classifyMiscDie(5)='smoke_or_repair'` | 菜单首行"施放烟雾" → 置 `sherman.smoked=true`；阶段①自动清除（德军由 AI `smoke` 执行，阶段④自动清除） |
| 舱盖切换 | 玩家点"舱盖"行 | 仅阶段② 可操作；车长阵亡 / 进入子阶段后锁定，提示"本回合舱盖已锁定" |

> **修复优先级建议**：当玩家同时承受炮塔受损 + 瘫痪 + 高着火程度时，一般推荐顺序为 **灭火 → 修复瘫痪 → 修复炮塔**（着火会每回合连锁伤害；瘫痪导致完全无法脱离危险格；炮塔受损仅影响输出）。

### 6.2 关键技术点

#### 6.2.1 六角格坐标
推荐采用 **Axial 坐标系**（q, r），渲染时转笛卡尔：
```ts
// pointy-top 六边形
const SIZE = 60;
function axialToWorld(q: number, r: number): Vec2 {
  const x = SIZE * Math.sqrt(3) * (q + r / 2);
  const y = SIZE * 1.5 * r;
  return new Vec2(x, y);
}
```
邻接 6 方向常量、距离公式（cube distance）封装在 `HexGrid.ts`。

#### 6.2.2 数据驱动任务
每关用 JSON 表示，包含：
- 地形矩阵（每格类型 + 是否树篱边）
- 初始单位放置（位置 + 朝向）
- 红色数字格（援军生成位）
- 行动表 / AI 表 / 事件表（覆盖默认值）
- 胜负条件（脚本表达式或枚举）

**复合目标 `destroy_kind_evac`**：在击毁 `objective.kind` 所指全部敌方单位后，谢尔曼须从 **`evacAt`** 六角格沿 **`evacExitDir`** 六向之一执行 **前进** 或 **后退**（后退的位移方向 = `facing+3`，须等于 `evacExitDir`），且 **邻格不在地图内**（无 `Tile`）时撤离成功并判胜；动画结束后谢尔曼坐标可落在网格外。存档字段 `shermanEvacuated` 由 `SaveLoad` 持久化。若配置 **`objective.kinds`**（`UnitKind[]`，非空），则须将所列 **每一种** kind 的敌方单位**全部**击毁后才允许撤离（与单独的 `kind` 二选一：有 `kinds` 时优先用 `kinds`，典型见任务 9：虎式 + IV 号，步兵不计入）。

**徒步类单位（步兵 / 军官，`isFootKind` / `isFootUnit`）**：`UnitKind` 中 `'infantry'`（步兵）与 `'officer'`（军官，任务 8 起）共享「徒步类」语义——`size = 0`，无装甲，仅可被机枪打死、不参与坦克 AI、不阻塞坦克叠格、不渲染装甲 / 装填面板。`types.ts` 导出的 `isFootKind(kind)` / `isFootUnit(unit)` 是这两类共有判定的统一入口；`Combat.canMGAttack`、`TurnEndEventApply.adjacent_infantry_fire`、`BattleScene` 的目标筛选 / 渲染分支等都通过它们进行兼容判定。**唯一例外**：`infantry_spawn` 事件 spawn 出来的单位 `kind` 始终为 `'infantry'`（不会复活军官），所以「按 `kind` 精确判定」的位置（如 `Objective.allEnemiesOfKindDestroyed(mission, 'officer')`）不能换成 helper。

**新单位 `'officer'`（军官）**：与 `'infantry'` 数值一致（见 `data/units.csv`），但 `kind` 独立——典型用法见任务 8（`mission_08.json`）：建筑里固定放置 1 只军官，objective 为 `{ type:"destroy_kind_evac", kind:"officer", evacAt, evacExitDir }`，与回合结束 5–6 spawn 出来的普通步兵互不混淆。UI 会在军官身周加红色光环、并在所在格绘制红色 hex 边框（`OFFICER_HALO_STROKE` / `OFFICER_TILE_STROKE`），与说明书原图「红色边框建筑里的德军步兵」一致。

**敌方掷骰出生（可选，`enemyStartByDice: true`）**

适用于「开局时敌方位置不完全固定」的关卡。地图上用 **`eid`**（1～6，全图**不重复**）标记**坦克等**的**黑格出生位**；用 **`rid`**（1～6，全图**不重复**）标记**步兵**的**红格（援军）出生位**。同一格可选 **`ef`**（0=E … 5=NE）表示该格上单位**初始 facing**（与掷骰放坦克 / 步兵均适用），其方向编号与**同格** `h` 的 6 位、以及 `HEX_DIRECTIONS[ef]` **完全一致**（自本格心指向第 `ef` 邻格心 = 经该边越境的六向）。树篱位 `h[i]=1` 与朝向 `ef` 均用**轴向**同一套索引 i，与 `HexGrid.hedgeFlagsFromMapJson` 及 `HEX_DIRECTIONS` 一致；`BattleScene` 绘制时经 `HEDGE_DRAW_EDGE_BY_AXIAL` 将轴向 i 映到 `drawHedgeEdge` 的**几何边**号。

- 关卡 JSON 中 **`enemyStartByDice: true`** 时，每个 **`enemies[]`** 条目可省略 `at` / `facing`，由运行时掷骰决定。
- **谢尔曼掷骰（可选，`shermanStartByDice: true`）**：与坦克共用地图上的 **eid 1..6** 黑格表，须同时 **`enemyStartByDice: true`**；`sherman` 可省略 **`at` / `facing`**，由 `MissionLoader` 在加载时**先于**德军无坐标单位掷 1d6 链式择空位（与坦克规则相同，占位后写入格上 `ef` 朝向）。典型任务见 `mission_11.json`（钢铁角斗士）。
- **`kind: infantry`**：**每名无 `at` 的步兵开局掷 1 颗 d6**，链式占用 **`rid = 骰点` → `rid+1` … 绕回 1** 的六角格（规则与下款相同，但查 **`rid` 表**，不是 `eid`）；地图上须配置至少一处 **`rid`**，否则加载失败。
- **坦克等非步兵**：**每名无 `at` 的单位开局掷 1 颗 d6**，首先尝试落在 **`eid` 等于骰点** 的六角格；若该格已被占用（含已放置的其它敌方单位，或谢尔曼初始格），则改试 **`eid = 骰点 + 1`**（6 的下一档为 1），再重复直至找到**既存在 `eid` 配置、又未被占用**的格；若连续 6 档仍无法落位则加载失败（配置错误）。
- 未配置对应编号的格在链式探测中跳过（例如关卡只配置了 `eid` 1、2，掷出 5 时会依次尝试 5→6→1→2…直到命中已配置且空闲的格；**`rid` 链同理**）。
- 实现见 `MissionLoader.loadMission(data, rng)`：需在创建任务后传入与整局共用的 **`RNG`**（如战斗场景种子），以便与后续随机流一并复现。

### 6.3 核心规则伪代码示例

```ts
// Combat.ts
export interface AttackParams {
  attacker: Unit;
  target: Unit;
  distance: number;
  hedgeCount: number;
  inBuilding: boolean;
  inSmoke: boolean;
  isHidden: boolean;
  fromBack: boolean;
  commanderHatchOpen: boolean;
}

export function resolveAttack(p: AttackParams, rng: RNG): AttackResult {
  // Step 1: hit
  const need = p.target.size + p.distance + p.hedgeCount
             + (p.inBuilding ? 1 : 0)
             + (p.inSmoke ? 1 : 0)
             + (p.isHidden ? 2 : 0)
             + (p.fromBack ? 1 : 0);
  const d1 = rng.d6(), d2 = rng.d6();
  const hitSum = d1 + d2;
  const isHit = hitSum > need;
  if (!isHit) return { hit: false };

  // Doubles + open hatch → commander dies
  const isDoubles = (d1 === d2);

  // Step 2: damage?
  const armor = pickArmor(p.target, p.attacker.facingRelative);
  const damageRoll = rng.d6();
  const causeDamage = damageRoll >= armor - p.attacker.penetration;
  if (!causeDamage) return { hit: true, doubles: isDoubles, damaged: false };

  // Step 3: damage type
  const typeRoll = rng.d6();
  const damageType = lookupDamageTable(p.target.faction, typeRoll);
  return { hit: true, doubles: isDoubles, damaged: true, damageType };
}
```

### 6.5 多语言与文案管理（lang.csv）

**硬规则**：**游戏里所有用户能看到的文本**（HUD、按钮、弹窗、浮字、状态面板、骰子托盘提示、伤害 / 乘员结算、胜负标语、攻击失败原因等）**都必须通过 `lang.csv` 读取，不允许在代码里硬编码中文 / 英文字符串。**同时游戏需支持运行时切换语言，默认显示中文。

#### 6.5.1 工程布局

| 文件 | 角色 |
|------|------|
| `data/lang.csv` | **文案主表**（策划直接维护）。UTF-8 + BOM，3 列：`key,zh,en` |
| `tools/buildLangDB.js` | CSV → TS 生成器。零依赖 Node 脚本 |
| `assets/scripts/core/LangDB.ts` | **自动生成**产物，不可手改（会被覆盖） |
| `assets/scripts/core/Lang.ts` | 运行时入口：`t()` / `setLang()` / `getLang()` |

#### 6.5.2 CSV 约定
- 3 列固定顺序：`key,zh,en`。三列都不允许为空，否则生成脚本拒绝通过。
- `key` 全局唯一，建议用点号分组命名：`btn.nextPhase`、`hud.movePhase`、`dmg.effect.fire`、`crew.role.1`。
- 值里允许 `{name}` 形式的占位符，调用 `t(key, { name: value })` 替换。
- 值里允许字面量 `\n`（反斜杠 + n），运行时自动展开为真正的换行，方便 Excel 编辑多行文案。
- 如果值里需要逗号或换行，按标准 CSV 用双引号包裹字段（`""` 转义内部引号）。

#### 6.5.3 运行时 API
```ts
import { t, setLang, getLang } from '../core/Lang';

label.string = t('btn.nextPhase');                             // 下一阶段
label.string = t('hud.movePhase', { n: 3, dice: '4/5' });      // 回合 3 | 移动阶段 | 骰子 4/5
label.string = t('preview.impossible', { n: 12 });             // 两行文案，\n 自动换行

setLang('en');   // 一行切到英文，下一次 t() 调用立即返回英文
```
- 找不到 key 时 `console.warn` 并原样返回 key，线上也能一眼看出缺漏。
- 默认 `currentLang = 'zh'`，语言切换可以绑定到主菜单的"设置"入口。

#### 6.5.4 策划工作流
1. 用 Excel 打开 `data/lang.csv` → 编辑 `zh` / `en` 两列（`key` 列别动）。
2. "另存为 CSV UTF-8（逗号分隔）" 覆盖原文件。
3. 在项目根目录执行 `node tools/buildLangDB.js`，自动重写 `LangDB.ts`。
4. 在 Cocos Creator 里预览即可看到新文案；无需重启编辑器。

#### 6.5.5 写代码的硬约束
- **绝不在 `.ts` 里硬编码中/英文字符串给任何 `label.string` / `button.text` / `spawnFloater()` 文案参数**。新增文案 → 先进 `lang.csv` → 跑生成器 → 再在代码里 `t('your.key')`。
- `core/` 层函数如果需要返回"原因类文本"（如 `canAttack().reason`），**只返回 i18n key**（参见 `Combat.AttackDenyReason`），由 UI 层统一 `t(reason)` 渲染。核心层与文案必须零耦合。
- 控制台日志 (`console.log` 等) 允许保留中英文硬编码——仅供开发者调试，不视为玩家可见文本。
- 编辑器 Inspector 的 `@property({ tooltip })` 同理：只是策划工具，不算玩家可见文本。

### 6.6 数值数据管理（CSV 驱动）

**硬规则**：**游戏里所有可调数值**（单位装甲 / 穿甲 / 体型、敌方 AI 行动表、伤害表、地形移动成本、骰子池大小、任务奖励、乘员加成……）**都必须配置到 `data/*.csv` 里，由生成脚本编译成 `assets/scripts/core/*DB.ts`，禁止把数字魔法常量散在代码中。**

目的：策划打开 Excel 改格子就能调参，不用等程序员改代码。

#### 6.6.1 现有 / 规划中的 CSV 表

| CSV | 生成器 | 输出 | 内容 |
|-----|--------|------|------|
| `data/units.csv` | `tools/buildUnitDB.js` | `core/UnitDB.ts` | 单位类型：体型 / 4 个装甲面 / 穿甲 |
| `data/lang.csv` | `tools/buildLangDB.js` | `core/LangDB.ts` | 所有玩家可见文本（见 6.5）|
| `data/player_action_table.csv` + `data/player_dice_pool.csv` | `tools/buildPlayerActionDB.js` | `core/PlayerActionDB.ts` | §3.6 玩家行动表（§3.6.2：1..6 + 对子 → A/B/C）+ §3.6.1 掷骰数（子阶段 × 地形基础 + 按阶段乘员 / 舱盖修正 + 上下限）|
| `data/enemy_ai_table.csv` + `data/enemy_ai_dice.csv` | `tools/buildEnemyAIDB.js` | `core/EnemyAIDB.ts` | §3.7 德军坦克 AI 表（列 × 骰面 → 主动作 / 备选动作）+ 每列骰数（road 4 / field 4 / mud 3 / damaged 2）|
| `data/terrain.csv`（规划）| `tools/buildTerrainDB.js` | `core/TerrainDB.ts` | 地形移动成本 / LoS 阻挡 / 射击修正 |

| `data/missions/*.json` | *(不走 CSV)* | *(直接 JSON)* | 单关地图 + 初始摆放；结构太复杂，继续用 JSON |

> 命名约定：CSV 文件统一放 `data/` 下，生成器统一放 `tools/build*DB.js`，输出统一到 `assets/scripts/core/*DB.ts` 并带"自动生成，请勿手改"注释头。

#### 6.6.2 生成器通用约定
1. **UTF-8 + BOM**：所有 CSV 强制 UTF-8 + BOM。生成器发现非 BOM 或 GBK 会自动转换并回写，保证 git diff 稳定、Excel 双击打开不乱码（参考 `buildUnitDB.js::readCsvSmart`）。
2. **强校验**：
   - 关键列（枚举类 key / 外键引用）必须落在预设集合里，否则生成器报错并**拒绝写文件**，不允许污染游戏代码。
   - 数值列必须是合法整数 / 浮点数，空值或非数字会被精确定位到 CSV 行号。
   - 唯一约束：key / 主键列不允许重复。
3. **零依赖**：只用 Node 18+ 内置 `fs` / `TextDecoder`，不引任何 npm 包；CI 环境和本地都能直接跑。
4. **稳定 diff**：生成的 TS 输出列顺序 / 对齐 / 注释格式固定，让 git diff 只反映真实的数值变动。

#### 6.6.3 代码侧读取约定
- 运行时所有数值查询**只走生成出来的 `*DB.ts` 导出函数**（如 `getUnitStats(kind)`、`t(key)`），不允许 `import` 原始 CSV。
- 不允许在 `.ts` 里写如 `const SHERMAN_ARMOR = 6;` 这种魔法数字；改成 `getUnitStats('sherman').armorFront`。
- 如果某个数值还没上 CSV（比如仍在玩法探索期），代码里写 `TODO(data-csv): 上表名.csv` 注释，列入下一次数据化任务。

#### 6.6.4 策划工作流
```
Excel 开 data/XX.csv → 改数值 → 另存为 CSV UTF-8 →
  node tools/buildXXDB.js → Cocos 预览即时生效
```

脚本失败时终端会打印具体到 CSV 行号的原因（`第 5 行 unitKind="..."：字段 armorFront="abc" 不是非负整数`），定位 1 秒搞定。

## 八、风险与对策

| 风险 | 影响 | 对策 |

## 九、首期可立即落地的工作清单（给 Cocos 新手）

1. **熟悉编辑器**：建好 `Battle.scene`，铺一张 9×9 的六角格预制体网格。
2. **数据先行**：手写 `mission_01.json`，包含地形矩阵 + 初始单位坐标。
3. **核心规则与视图分离**：所有规则写在 `core/`，纯 TypeScript 不引用 `cc.*`，方便单元测试。
4. **小步迭代**：先做"摇骰 → 显示骰子 → 玩家点击前进"的最小回路，再逐步加规则。
5. **使用 Git 分支**：每个 Milestone 一个分支，主分支始终可运行。
