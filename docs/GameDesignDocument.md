# 《孤胆谢尔曼》手机游戏策划案

> 基于同名桌游规则改编的单人战棋手游  
> 引擎：Cocos Creator 3.8.8  
> 平台：iOS / Android（横屏为主，竖屏适配）  
> 类型：单机回合制 / 战棋 / Roguelite  
> 版本：v0.1（首版立项稿）

---

## 一、项目概述

### 1.1 游戏名称
- 中文：《孤胆谢尔曼》
- 英文（暂定）：**Lone Sherman: Tactical Tank Commander**

### 1.2 游戏背景
玩家在二战法国战场指挥一辆美军 M4 谢尔曼坦克及其乘员，独自面对由 AI 驱动的德军坦克与步兵，完成 12 个不同主题的战术任务。游戏不追求绝对历史精确，强调"剧情驱动 + 战术决策 + 骰运紧张感"的体验。

### 1.3 核心卖点
1. **真正的单人战棋**：所有敌方行动由确定性规则表（AI 表）驱动，不需要联网、不需要对手。
2. **掷骰即战场**：以 5–7 颗骰子（虚拟）作为唯一行动资源，每一次摇骰都是一次决策。
3. **乘员 RPG 化**：5 名乘员（车长、炮手、装填手、驾驶员、副驾驶）各自影响掷骰数量，可阵亡、可补员，单局养成。
4. **纸面规则数字化**：把桌游中"找表 → 修正 → 比较"的繁琐流程自动化，留下决策乐趣。
5. **章节任务 + 战役模式**：第一章"欧洲战场"包含 12 关；后续章节可继续追加任务，可单关游玩，也可串联成战役（保留炮塔受损、痛痪、着火程度跨关延续）。

### 1.4 目标用户
- 战棋 / 桌游爱好者（Memoir'44、Combat Commander、Patton's Best 玩家）
- 喜欢硬核单机战术、Roguelite 关卡的轻度核心玩家
- 二战题材爱好者

### 1.5 商业模式
- 买断制（建议 ¥18–¥30）+ 任务包 DLC（后续追加 6–12 关）
- 不含内购、不含广告，主打"纯净战棋"体验

## 二、核心玩法循环

### 2.1 单关流程

```
进入任务 → 设置阶段（自动放置敌方）
   ↓
进入回合循环（8 阶段）：
  ① 移除玩家阵营烟雾（玩家控制坦克 + 队友）
  ② 决定车长舱盖位置（开/关）
  ③ 玩家控制坦克行动（移动 / 攻击 / 杂项）—— 玩家核心决策
  ④ 移除德军烟雾
  ⑤ 着火程度检定
  ⑥ 友军坦克 AI 行动（队友，若本关存在）
  ⑦ 德军坦克 AI 行动
  ⑧ 回合结束事件（援军、地雷、斯图卡等）
   ↓
判定胜负（达成任务目标 / 玩家控制坦克被摧毁 / 全员阵亡）
   ↓
战后结算（乘员补充、状态延续）→ 下一任务 / 返回主菜单
```

### 2.2 战役流程
- 第一章 12 关任务可顺序通关，也可随机抽选构筑"战役"；后续章节沿用同一章节/关卡数据结构扩展。
- 关与关之间允许"恢复"：补充 1 名阵亡乘员（玩家选择）。
- **跨关延续**：炮塔受损、痛痪状态、着火程度、装填状态全部保留，制造决策连锁。

### 2.3 一次"玩家行动阶段"（玩家核心决策）拆解
```
① 由**当前子阶段（移动 / 攻击 / 杂项）**、**玩家控制坦克当前格地形**与**乘员存活 / 舱盖**等，按 §3.6.1 表格规则自动计算本阶段可掷骰子数（再经配置的上下限钳制）
② 玩家点击"摇骰"按钮，按该数量播放骰子动画落定
③ 玩家把骰子按顺序（A 移动 / B 攻击 / C 杂项）拖入行动槽
   - A 与 B 顺序可互换（玩家选择"先移动后攻击"或"先攻击后移动"）
   - C 必须最后执行
④ 每个行动按对应规则结算（移动动画、命中判定、装填、烟雾等）
⑤ 阶段结束，进入下一阶段
```

> **控制权约定**：玩家每关只直接控制 1 辆己方坦克。当前版本默认是 M4 谢尔曼；后续可通过关卡或战役配置替换为其它可控坦克型号。队友坦克属于友军单位，不能被玩家手动控制，只在阶段⑥（着火程度检定之后、德军坦克行动之前）按 AI 规则自动行动。

> **设计要点**：要在手机端把"骰子拖动 + 行动槽匹配"做得直观，是 UX 最大挑战之一。详见第六章。

> **乘员死亡机制**：当主角受伤的伤害结果为"阵亡"时，掷 1d6 决定哪位乘员死亡（编号 1–5）。乘员可被同一伤害类型再次命中（已死的乘员需重新掷骰）。

#### 3.1.2 友军单位

- 友军单位与玩家控制单位同属玩家阵营，可作为任务初始单位、援军或剧情单位出现。
- 玩家**不能**直接控制队友坦克；队友坦克在着火程度检定之后、德军坦克行动之前执行 AI 行动（见 §2.1 阶段⑥）。
- 队友坦克的行动逻辑复用 §3.7 的坦克 AI 行动表与动作规则，区别只在于：
  - 阵营为友军；
  - 攻击目标与朝向判断中的"敌方坦克"指德军坦克；
  - 不参与玩家行动骰、乘员指令与玩家手动选择。
- 队友坦克可使用与德军坦克相同的基础字段（`kind`、`faction`、`at`、`facing`、`damaged`、`hidden`、`smoked` 等），以便实现层复用同一套规则结算。

### 3.2 地形系统

| 地形 | 颜色/示意 | 进入限制 | 视线 | 战术效果 |
|------|----------|---------|------|---------|
| 公路 | 灰色 | 全单位可入 | 不阻挡 | 谢尔曼/敌坦克移动阶段 +1 骰 |
| 田地 | 浅黄绿 | 全单位可入 | 不阻挡 | — |
| 泥地 | 棕色 | 全单位可入 | 不阻挡 | 谢尔曼/敌坦克移动阶段 -1 骰；痛痪状态下若被推入将无法行动 |
| 林地 | 深绿 | **坦克不可入**；可作为视线起止 | **阻挡视线** | 步兵可入 |
| 树篱 | 绿色细带 | 沿格边延伸；不影响进入 | 沿线提供掩护 | 每穿过 1 个树篱命中所需 +1；**紧挨攻击者的树篱不计**（详见 §3.4 Step 1） |
| 水域 | 蓝色 | **任何单位不可入**（除非该格叠加桥梁，详见下文） | 不阻挡视线 | — |

**桥梁（叠加在水域格上，非独立地形类型）**

- 桥梁**不是**与上表并列的一种地形，仅能**叠加**在 `t === "w"` 的水域格上。配置桥梁的水域格相当于"在水面上架了一座桥"，**坦克 / 卡车可进入**；不带桥梁的水域格仍然完全不可入。
- **骰子规则与公路相同**：桥梁格的移动 / 攻击 / 杂项基础骰数全部按公路读取，移动力消耗也按公路计（与基底"水域"完全脱钩，避免出现"+0 颗骰、不可入但又能动"的死锁）。
- **「按公路触发」的规则一律视桥梁为公路**：除骰子 / 移动力外，所有以「站在公路格」为前置条件的事件——典型如 §3.8 / 任务事件表中的 **`road_mine` 公路地雷**——也同样在桥梁格上触发；实现层统一通过 `effectiveDiceTerrain(tile)` 把"水域+桥梁"折算为 `'road'` 后再判定，避免出现"桥上吃骰加成、却躲掉地雷"的不一致。
- **必须配置两端方向**（关卡 JSON：`TileDef.br = [a, b]`，`a, b ∈ 0..5` 且不重复）。这两个方向定义了桥梁向**外**贯通的两条边；其余 4 条边视为水面，车辆不可越。
- **越境规则**：车辆（坦克 / 卡车）只能从「桥梁两端方向之一」**驶入**或**驶出**桥梁格——
  - 进入：从相邻格 A → 桥梁格 B 时，必须满足 `dir(B → A) ∈ B.br`（即 A 落在 B 的某一桥梁端方向上）；
  - 离开：从桥梁格 B → 相邻格 C 时，必须满足 `dir(B → C) ∈ B.br`（即 C 落在 B 的某一桥梁端方向上）。
  - 两个相邻桥梁格相连时，"出"与"入"两条约束同时生效——两端必须**同时对齐**才允许跨越（即两座桥的桥端方向都覆盖那条物理边）。
- 视线 / 树篱：桥梁不影响视线（与水域一致：不阻挡），桥梁本身不附带树篱修正。
- 撤离 / 驶出：若桥梁格的某一端方向指向地图外，车辆可从该端**驶离地图**（与公路出口相同），此约束同样适用于谢尔曼撤离 (`destroy_kind_evac`) 和卡车末格 `truckPath.exitDir`：方向必须落在 `br` 端之一，否则会被边向校验拒绝。

**建筑（叠加在基底格上，非独立地形类型）**

- 建筑**不是**与上表并列的一种地形，而是在**公路 / 田地 / 泥地**等基底格上叠加的标记（美术可用房屋图案表示）。
- 基底为**田地**或**泥地**且带建筑时，在规则与任务表述中一并视为**农场**；基底为**公路**且带建筑时，视为**村庄**。
- 进入与掩护：**坦克可以进入**带建筑的格子；对**被攻击目标**而言，目标格有建筑仍提供掩护（§3.4 Step 1「目标位于建筑格 +1」）。
- 视线：从建筑**以外**的格子发出的射线，若路径上存在**中间格**（非起点、非终点）为带建筑格，则**被遮挡**；建筑格作为视线的**起点**或**终点**时，该格上的建筑**不**截断该条视线（建筑内单位可由此格向外开火）。林地仍阻挡视线；**水域不再阻挡视线**（仅保留「任何单位不可入」的进入限制）；树篱不减损视线、仅按 §3.4 修正命中（见上表）。

### 3.3 朝向、视线与射程
- 朝向 = 坦克车体（**不是炮塔**）面对的 6 个格边方向之一。
- 视线沿 6 个直线方向无限延伸（射程不限）：**穿过**路径上的林地，或**作为中间格**（非起止格）的带建筑格时，视线被阻挡；**水域不阻挡视线**；树篱不截断视线，但按 §3.4 修正命中。建筑格作为视线**起点**（例如建筑格内的坦克向外射击）或**终点**时，该格不视为沿该方向的视线阻挡物。
- 攻击方向决定使用哪一面装甲值（前 / 前侧 / 后侧 / 后），是核心战术深度。
- **方向圈**：以坦克为中心的 6 格分为前 / 前侧（×2）/ 后侧（×2）/ 后。
- **装甲面判定（实现约定）**：取从**目标格心**指向**攻击者格心**的六向射线（与主炮视线「攻击者与目标共线」一致；代码里与 `directionTo(目标, 攻击者)` 对齐），再相对车体 `facing` 求差得到前 / 前侧 / 后侧 / 后。若任务 JSON 里 `facing` 与美术车体「前」不一致，会出现「体感前后反了」——应改关卡朝向而非改掷骰公式。

#### 3.3.1 战争迷雾与玩家视野

**游戏模式与启用范围**

- 玩家在主界面进入关卡前选择“经典”或“硬核”模式；选择会作为菜单偏好保存，并通过 `GameSession.gameMode` 传入战斗。新玩家默认使用经典模式。
- 经典模式关闭战争迷雾，保持全图可见；硬核模式开启战争迷雾，并执行本节全部玩家信息隐藏规则。
- 战争迷雾不再由具体关卡 JSON 配置，任务数据不包含 `fogOfWar`。模式差异统一登记在 `GameMode` 配置表，后续乘员加成规则、AI 行动表等模式差异也从该入口选择各自配置，避免在关卡中重复声明难度规则。
- 战斗存档保存创建该战局时的模式；“继续游戏”恢复存档模式，不允许用主界面当前选择临时改变既有战局规则。旧版未记录模式的存档按经典模式恢复。
- 经典模式不创建迷雾遮罩，也不执行下述玩家信息隐藏规则，画面与无迷雾版本一致。

**术语与基础表现**

- `当前可见格`：玩家控制的主角车辆在当前坐标、车体朝向及车长舱盖状态下，此刻能够获取视野的地图格。
- `迷雾格`：存在于地图中、但不属于当前可见格集合的格子。迷雾只表示**当前不可见**，不保留永久探索结果；车辆离开或转向后，失去视野的格子会重新进入迷雾。
- 迷雾格覆盖一层统一的**灰色半透明遮罩**。遮罩按单个六角格裁切，不能溢出到相邻格；建议初始透明度为 55%，最终数值以真机可读性测试为准。
- 遮罩位于地形、地形装饰、单位、出生位/援军位标记及地图内状态标记之上；迷雾内的敌军、友军、标记、**地图内**单位动画/特效和单位交互热点均不显示，不能从迷雾中选择攻击目标。迷雾格本身仍可点击并打开格子信息界面，但单位信息栏不读取或显示实际单位，统一显示“当前格单位状态未知”。玩家自己的主角车辆所在格始终可见。下方行动骰子属于全局流程反馈，不受此条限制。
- 战争迷雾是**玩家信息规则**，不改变地形、单位位置、敌方 AI 感知、移动合法性、回合结束事件或胜负条件的内部结算；只有需要玩家选择目标的行动，才必须把目标限制为当前可见单位。

**迷雾内单位的行动表现**

- 单位当前所在格没有玩家视野时，隐藏该单位本体及其地图内状态标记。轮到该单位行动时，同时隐藏跟随单位显示的行动提示框，不在迷雾格上暴露单位位置或本次动作文字。
- 无论行动单位是否可见，屏幕下方的行动骰子动画均按正常流程播放；掷骰、选骰和骰面结果的节奏不因战争迷雾缩短或跳过。
- 不可见单位执行移动或转向时，规则结算及行动时长保持正常，并正常播放对应的移动音效。单位本体在渲染层始终按正常动画绘制，但单位层只允许当前可见六角区域内的像素通过；位于迷雾下的单位像素会被完全裁掉，不会透过半透明地形遮罩显现。
- 不可见单位从迷雾格移动到当前可见格时，不等待抵达目标格心才突然显示：持续移动的单位按可见六角边界逐像素露出，表现为车体先露出一部分、再完全驶出迷雾。单位状态标记、名称和行动提示框仍在正式抵达目标格后恢复；若本次行动后还有后续动作，后续动作全部按可见单位的正常表现执行。可见单位移入迷雾格时采用相反规则，由六角边界逐步裁掉车体，抵达后完全隐藏。
- 攻击动作开始时，以攻击单位当前所在格是否可见决定本次攻击表现。攻击单位原本不可见时，仍跳过该次攻击的命中、击穿和伤害等判定界面，但开火会立即暴露攻击者位置：攻击者所在格临时移除迷雾，并显示单位本体与开火表现。
- 临时揭示只覆盖攻击者当前所在的 1 格，不扩展相邻视野，也不改变永久/常规可见格集合；敌方与友方单位使用同一规则。
- 不可见单位发动攻击时，正常播放该武器配置的开炮/攻击音效；拥有独立炮塔的单位在显形前直接把炮塔朝向本次射击目标，主炮等有炮口表现的武器同时显示正确方向的炮口火焰。临时揭示持续 **0.9 秒**；开火动作结束后撤销临时揭示，若该格仍不属于玩家常规视野，则立即恢复迷雾并重新隐藏攻击者。
- 后台仍完整执行所有掷骰、修正、伤害、副作用及状态变更，攻击结果与无迷雾时完全一致。若被攻击单位位于当前可见格，则正常显示该单位受到攻击的地图内结果反馈，包括 `MISS`、跳弹/未击穿、命中、受损、起火、乘员伤亡、瘫痪或被击毁等实际结算结果；只省略攻击判定界面。若被攻击单位也位于迷雾格，则不显示其地图内结果反馈，但结果仍在后台生效。
- 若攻击单位在本回合较早的移动中进入当前可见格，则其后发起的攻击不再适用隐藏攻击规则，恢复完整的行动提示、判定界面、攻击动画和结果反馈。

**视野阻挡物**

- 下列格子会阻挡其后的视野：林地（`terrain === 'forest'`）、岩石高地（`terrain === 'rocky'`）、带建筑格（`hasBuilding === true`）。
- 阻挡格**自身仍然可见**；只有射线中位于它之后的格子被遮挡。观察起点格即使带建筑，也不阻挡从本格向外观察。
- 水域、深水、桥梁、树篱、防波堤、公路、田地、泥地、海滩、Pacific 树木和机场均不阻挡玩家视野；其中 Pacific 树木是可进入地形，不等同于欧洲林地。

**视野范围属性**

- 玩家主角车辆拥有当前视野范围属性 `visionRange: number`，初始值为 **4**。该值按非负整数格处理。
- `visionRange` 是运行时状态，不是战争迷雾算法中的固定常量；后续天气、任务事件、车辆状态或其它系统可以临时或持续修改当前值，视野刷新时立即按新值重算。
- 关卡 `sherman.visionRange` 可覆盖初始值；缺省时由任务加载器写入 4。存档保存当前值，读档后不重置为初始值。
- 无论车长是否打开舱盖，任何由车体朝向获得的直线视野最大距离都等于当前 `visionRange`，不得越过该上限。

**车长未打开舱盖**

- 主角车辆所在格及其六个相邻格始终可见。
- 额外获得炮塔当前朝向 `turretFacing` 的一条六角直线视野，从第 1 格开始最多延伸当前 `visionRange` 格；炮塔朝向与车体朝向 `facing` 可不同。
- 炮塔射线遇到首个建筑、林地或岩石高地时，该阻挡格自身可见，射线在此终止；达到当前视野范围上限、走出地图或遇到地图空洞时同样终止。

**车长打开舱盖**

- 基础观察半径等于当前 `visionRange`：以主角车辆所在格为中心，六角距离 `<= visionRange` 的每个地图格都作为候选格；候选格只有在起点到目标格之间不存在阻挡格时才可见。初始状态下 `visionRange = 4`，因此初始观察半径为 4。
- 目标格自身是林地、岩石高地或带建筑格时仍然可见；判定只检查起点与目标之间的中间格。
- 除范围视野外，仍按同一属性计算“车体正前方直线视野”；正前方视野不得超过当前 `visionRange`，阻挡与终止规则同未开舱状态。
- 若车长阵亡，舱盖不能保持打开，玩家视野按“未打开舱盖”计算。

**非直线遮挡判定（开舱半径视野）**

1. 对每个满足距离条件的候选格，从**玩家主角车辆所在格的中心点**向**候选格的中心点**绘制一条几何线段。坐标示例统一写作 `{col,row}`。
2. 依次检查这条中心点连线经过的中间六角格；“经过”是指线段穿过该格实际绘制的六边形区域，而不是按六角格最短路径、邻接寻路或泛洪结果推测。
3. 忽略起点格与目标格。只要中心点连线经过任意一个中间视野阻挡格（建筑、林地或岩石高地），目标格就被该阻挡格遮挡；目标格进入迷雾。
4. 不再采用“两条等价六角路径中任选一条畅通即可看见”的宽松拐角规则。视野只有这一条中心点几何连线，不能通过另一条邻接路径绕过阻挡格。
5. 示例：玩家 Sherman 位于 `{2,3}` 的泥地，`{4,1}` 为带建筑格。由 `{2,3}` 中心点连接到 `{5,0}`、`{6,0}` 中心点的视线经过 `{4,1}` 建筑格，因此 `{4,1}` 自身可见，而其后的 `{5,0}` 与 `{6,0}` 均被遮挡。
6. 该几何遮挡判定先用于判断候选格是否可见；候选格仍须满足“开舱时六角距离不超过当前 `visionRange`”或“正前方距离不超过当前 `visionRange`”等条件，不会因为连线无遮挡而获得额外视野范围。

**刷新时机与状态**

- 进入关卡、读取存档后立即计算一次视野。
- 主角车辆每次前进、后退或位置被事件改变后刷新；每次车体转向完成后刷新。
- 阶段②点击“打开舱盖/关闭舱盖”后立即刷新；车长阵亡导致舱盖关闭时同样立即刷新。刷新与对应移动/转向动画同步，避免终点信息提前暴露。
- 天气、任务事件或其它系统改变当前 `visionRange` 后立即刷新。
- 当前可见格集合是运行时派生数据，不作为存档真值保存；读档后根据主角位置、朝向、舱盖、乘员状态和当前 `visionRange` 重算。

|------|----------|

| 迷雾内单位移动至可见格 | 单位始终绘制，并按可见六角边界逐像素露出，呈现车体从部分露出到完全驶出迷雾的连续效果；抵达格心后恢复状态、名称、行动提示框及后续动作的正常表现 |

### 3.4 战斗系统（攻击 = 三步走）

#### Step 1：能否命中？（掷 2d6）
计算"命中所需点数"：

```
命中所需 = 体型 + 距离（六格距离）
         + Σ 树篱数 (每穿过 1 个 +1；紧挨攻击者的树篱不计)
         + 目标位于建筑格 (+1)
         + 目标有烟雾掩护 (+1)
         + 目标处于隐蔽状态 (+2)
         + 目标在攻击方背后 (+1)
```

掷 2d6，**两骰之和 > 命中所需** → 命中。  
**额外规则（欧洲 / Pacific 均适用）**：敌方攻击玩家控制 Sherman 时，若命中判定 2d6 掷出 2 个相同点数（"对子"）、且本次攻击命中成功、且车长处于"打开舱盖"位置 → 车长立即阵亡；未命中时即便掷出对子也不会触发，命中但未击穿 / 弹开仍会触发。
**表现规则（欧洲 / Pacific 均适用）**：任何攻击（机枪 / 主炮，玩家 / 友军 / 敌方）若命中判定失败，均在目标位置显示 MISS；机枪攻击步兵在命中不可能成功时跳过骰子动画，但仍播放机枪音效并显示 MISS。
**步兵攻击例外（欧洲 / Pacific 均适用）**：步兵攻击坦克时无视目标的隐蔽状态；即使命中目标处于 `hidden=true`，也不应用 "目标处于隐蔽状态 +2" 修正。

> **"紧挨攻击者的树篱不计"细则**（实现：`HexMap.countHedgesAlong`）：
>
> 路径上的某段树篱若**与攻击者紧挨**，则不计入命中所需。"紧挨攻击者"指：
> 1. 该树篱与攻击者**处于同一格子**（即攻击者格 `hedges[dir(攻击者→第一邻格)]` 命中）；
> 2. 或该树篱所处格子**与攻击者相邻**且**树篱方向为攻击者所在格**（即第一邻格 `hedges[dir(第一邻格→攻击者)]` 命中）。
>
> 这两种写法对应**同一条物理边**（攻击者射出第一步穿过的那一段），实现上从路径第二条边起累计、并对 `i ≥ 1` 的每条边检查双侧编码（`a.hedges[dirAB] || b.hedges[dirBA]`），命中计 1 不重复累加。
> 直觉解释：自家篱笆一侧是熟悉地形，开火时不会被自家树篱遮挡视线，因此不增加难度。

#### Step 2：造成伤害？（欧洲战场：掷 1d6）
```
d6 ≥ 目标对应面装甲 - 攻击方穿甲值  → 造成伤害
```
对应面取决于攻击方向（前/前侧/后侧/后）。

仅在**硬核模式**下，攻击方单位使用 `data/units.csv` 中配置的 `effectiveRange`（有效射程）：目标距离不超过有效射程时使用基础穿甲值；每超出 1 格，本次攻击的穿甲值减 1，最低为 0。衰减值只用于当前攻击及其预览，不修改单位的基础 `penetration`。经典模式始终使用基础穿甲值，格子详情也仅在硬核模式显示有效射程属性。

> **章节覆盖**：上式是第一章欧洲战场的击穿规则。第二章太平洋战场改用 2d6 击穿判定，见 §4.2.4；因此同名坦克在不同战场可以拥有不同装甲数值配置，不能用欧洲数值直接套入 Pacific。

#### Step 3：怎样的伤害？（掷 1d6 查表）

| d6 | 主角受伤 | 其他坦克受伤 |
|---|---|---|
| 1 | 被摧毁 | 受损 → 被摧毁 |
| 2 | 阵亡检定（见乘员表） | 受损 → 被摧毁 |
| 3–4 | 着火 / 着火程度 +1 | 受损 → 被摧毁 |
| 5 | 炮塔受损 | 被摧毁 |
| 6 | 痛痪 | 被摧毁 |

> 主角"阵亡检定"（欧洲 / Pacific 均适用）：再掷 1d6（仅 1 次）映射到乘员 1–5；若掷出 6 且车长处于打开舱盖 → 车长阵亡。
>
> 友军队友坦克被击穿时使用"其他坦克受伤"列，即使它与主角是同一坦克型号或同一美术资源，也不触发主角专属的乘员、着火程度、炮塔受损、痛痪结果。

### 3.5 着火 / 修复 / 状态系统

谢尔曼与敌方坦克在战斗中会累积可被再次解除的持久状态。下文按"总表 → 视觉 → 回合内检定 → 解除逻辑 → MVP 占位"五步展开，便于策划 / 程序 / 美术三方对齐。

#### 3.5.1 状态总表

| 状态 | 代码字段 | 触发 | 持续效果 | 解除方式 |
|------|---------|------|----------|----------|
| 着火程度 | `fireLevel: number`（0..∞，仅玩家控制坦克） | 玩家控制坦克受"着火"伤害（§3.4 Step 3，d6 = 3 或 4），每次 +1 | 阶段⑤"着火程度检定"：掷 N = fireLevel 颗 d6，但只取其中点数最小的 1 颗，按 §3.4 Step 3 主角受伤列结算 1 次；数值也是"火苗烈度"的视觉强度依据 | 灭火行动（杂项 6 点）-1，最低 0；`fireLevel` 归 0 时橙色外观立即回归阵营色 |
| 受损（其他坦克） | `damaged: boolean` | 其他坦克受伤列 d6 = 1..4；友军队友坦克同样适用 | 下次击穿直接升级为"摧毁"（MVP 语义） | 不可解除 |
| 炮塔受损 | `turretDamaged: boolean` | 伤害结果 = "炮塔受损"（§3.4 主角受伤列 d6 = 5） | 不能使用主炮射击 —— 攻击阶段 5 / 6 点（主炮骰）与杂项 1 点"炮手主炮"选项被禁用；右侧状态面板"炮塔"行显示 **受损** | 修复行动（杂项 4 点 或 5 点"烟/修"选"修"） |
| 瘫痪 | `paralyzed: boolean` | 伤害结果 = "瘫痪"（§3.4 主角受伤列 d6 = 6） | 不能前进 / 转向 / 后退 —— 移动阶段所有骰面动作一律失败；状态面板"机动"行显示 **瘫痪**；若后续被推入泥地则完全无法行动（§3.2 规则特例） | 修复行动（杂项 4 点 或 5 点二选一） |

#### 3.5.3 回合流程中的状态检定

> 以下顺序对应 §2.1 单关回合循环 8 阶段，只列出与本节状态有关的步骤：

- **阶段① 移除玩家阵营烟雾**：若玩家控制坦克或队友坦克 `smoked=true`，开局置 false。
- **阶段② 决定舱盖位置**：玩家每回合最多点击一次“打开舱盖/关闭舱盖”；点击后按钮立即灰显并锁定，进入后续子阶段后继续保持锁定，下一玩家回合开始时恢复可用。
- **阶段③ 玩家控制坦克行动**：
  - 移动 / 攻击 / 杂项 子阶段内凡受状态约束的动作（炮塔受损→禁主炮、瘫痪→禁移动…）在骰子托盘对应面直接表现为"无可用动作"。
  - 隐蔽与前进 / 转向 / 后退互斥：一旦消耗移动动作立即清除 `hidden`（同条规则在阶段⑥友军坦克 AI、阶段⑦德军坦克 AI 的转向 / 前进 / 后退动作上同样生效；见 §3.5.1 解除方式列）。
- **阶段④ 移除德军烟雾**：遍历所有存活敌方，`smoked` 统一置 false。
- **阶段⑤ 着火程度检定**：若主角 `fireLevel > 0`，掷 N = `fireLevel` 颗 d6，只取其中点数最小的 1 颗，按 §3.4 Step 3 的主角受伤列走 1 次伤害结算（可能触发"摧毁 / 瘫痪 / 炮塔受损 / 阵亡检定 / 起火 +1"）。**该阶段是关卡核心压力来源之一，迫使玩家在 A / B / C 骰分配上为"灭火"骰留额度。**
- **阶段⑥ 友军坦克 AI 行动**：队友坦克按 §3.7 共用 AI 表执行；`smoke` / `repair` 动作同样消费骰并修改对应状态。
- **阶段⑦ 德军坦克 AI 行动**：AI 表 `smoke` / `repair` 动作同样消费骰并修改对应状态。
- **阶段⑧ 回合结束事件**：结果亦可能触发"着火"、"瘫痪"等（见 §3.8）。

> **顺序约束**：A、B 可互换顺序，C 必须最后执行。
>
> **对子（doubles）触发方式**：只要本阶段托盘中存在两颗点数相同且均未使用的骰，点击其中任意一颗即在动作菜单末尾追加对应 A / B / C 列的对子动作项；选中后一次性消耗两颗。规则：
> - **A 移动**：普通单骰的前进 / 后退 / 转向不受驾驶员、副驾驶阵亡限制；乘员存活只改变本阶段骰数。`驾驶员前进（+同点骰）` 需驾驶员存活，`副驾驶 ↻ 60°（+同点骰）` / `副驾驶 ↺ 60°（+同点骰）` 需副驾驶存活；瘫痪时相关动作失败（骰不消耗，浮字提示）。
> - **B 攻击**：普通单骰装填与主炮射击不受装填手、炮手阵亡限制，B 列 3/4 点机枪也不受乘员阵亡限制；乘员存活只改变本阶段骰数。仅 `装填手装填（+同点骰）` 需装填手存活，`炮手主炮射击（+同点骰）` 需炮手存活。炮塔受损仍会禁止所有主炮射击。
> - **硬核模式主炮选格**：普通主炮骰、`炮手主炮射击（+同点骰）`、精确射击及杂项阶段的炮手主炮均不以 `loaded` 作为动作菜单门槛。已装填并点击当前可见的合法敌军时，沿用原开炮流程，消耗行动骰与炮弹。仅在**关舱状态**下，点击六向直线且六角距离不超过 `visionRange` 的迷雾格时，消耗对应行动骰并把炮塔旋转到该方向，不播放开炮、不进行攻击判定、不改变 `loaded`；炮塔转到位后立即按新 `turretFacing` 刷新关舱视野。选定主炮行动后，所有可执行该转向的迷雾格显示眼睛标志。关舱时点击非六向直线格无效且不消耗骰，提示“不能转向该方向”；直线迷雾格超出 `visionRange` 时同样无效且不消耗骰。**开舱状态**下点击任何迷雾格均不执行炮塔转向、不消耗骰，仍按原交互打开格子详情界面。
> - **硬核模式精确射击**：仅在攻击阶段，若当前点击的主炮射击骰（5 / 6 点）存在另一颗未使用的同点主炮射击骰，动作菜单额外显示“精确射击”。该动作要求炮手存活，选择后只能点击当前可见的合法敌军目标进行攻击，不能用于点击迷雾格旋转炮塔，也不显示炮塔侦察眼睛标志；开火时消耗主骰与同点搭档共 2 颗骰，主炮装填、合法目标、穿甲、伤害与普通主炮完全相同，唯一差异是最终“命中所需点数”在原公式基础上 **-2**。表现上，炮塔完成对准目标后保持静止 **0.5 秒**，再播放开火音效与炮口火焰并进入后续骰子表现，以明确表达瞄准过程。经典模式不显示该选项。
> - **C 杂项**：岗位动作受对应乘员存活限制：装填手装填需装填手存活、炮手主炮射击需炮手存活、驾驶员转向 / 前进需驾驶员存活、副驾驶机枪需副驾驶存活。`进入隐蔽（+同点骰）` 消耗两颗同点骰并置 `sherman.hidden=true`；任意后续移动动作（转向 / 前进 / 后退）立即清除。硬核模式下，同点骰菜单额外提供 `关闭舱盖（+同点骰）`；仅车长存活且舱盖打开时可用，执行后消耗两颗同点骰并令 `hatchOpen=false`，立即按关舱视野刷新战争迷雾。经典模式不提供该动作。
>
> **UI 交互**：若一颗骰原本有"无分支直接执行"的动作（如移动 5/6 前进、移动 1 后退、攻击 1/2 装填、杂项 6 灭火），当它同时拥有同点搭档时，不再跳过菜单，仍会弹出 popover，以便玩家在"单骰动作"与"对子动作"之间做选择。玩家已选定主炮开火但尚未消耗该骰时，点击任意其它骰子会先取消当前主炮选择态及地图提示，再处理新点击骰子的动作，避免残留选定状态导致误操作。

**当前目标定义**：AI 规则中旧文案凡写作"谢尔曼"或"玩家单位"的位置，一律替换为"当前目标"。当前目标不再局限于坦克；敌对坦克、重炮、反坦克炮和步兵都可以成为当前目标。
1. 先按目标优先级筛选：**任务目标 ＞ 坦克 = 重炮 = 反坦克炮 ＞ 步兵**。该优先级高于距离优先级；只在同一优先级内比较距离。
2. 在最高优先级的候选目标中，选择距离行动单位最近者。
3. 若同优先级且距离相同，则从并列目标中随机选择 1 个。

> **任务目标与阵营例子**：德军单位行动时，玩家控制坦克是其任务目标；友军队友行动时，关卡胜利条件指定摧毁的单位为任务目标。若关卡要求清空全部敌军，则所有存活敌军都属于任务目标。卡车等未列入常规优先级的单位，只有被关卡胜利条件明确指定时才可作为任务目标。

**主炮射击目标修正**：AI 表执行 `射击 / shoot` 动作时，先以 §3.4 `canAttack` 规则筛选所有视野内可由主炮攻击的合法敌方目标（必须在 6 条直线上，且视线未被林地 / 中间建筑等阻挡；炮塔受损等状态同样会使射击动作不可执行）。
1. 若当前目标不在视野内或被阻挡，AI 会继续寻找其它视野内合法目标。
2. 若存在合法目标，仍先应用当前目标优先级，再在同一优先级内选择距离行动单位最近者。
3. 若多个合法目标优先级与距离均相同，从并列目标中随机选择 1 个。
4. 经典模式下，若没有任何合法主炮目标，则本次 `shoot` 视为不可执行，按 AI 表备选动作或空转处理。
5. 硬核模式下，敌方或友方**坦克**的 `shoot` 动作没有合法主炮目标时，立即改用玩家 Sherman 同款机枪规则重新筛选敌对步兵目标：复用 `canMGAttack` 的步兵类型、非同格、直线与视线条件，以及同一套 1d6 命中阈值、正面 -1、命中即击毙和不可能命中处理。若有多个合法步兵目标，仍优先任务目标，再在同级目标中选择最近者，同级同距随机；找到目标即以本次 `shoot` 骰执行机枪射击。仅当主炮和机枪均无合法目标时，才继续检查 AI 表备选动作或空转。反坦克炮、重炮、卡车和步兵不获得此回退。

**硬核模式 AI 精确射击**：敌方或友方坦克执行 `shoot` 且已经选出合法主炮目标后，按以下顺序判断：① 当前行动骰之外还有一颗未使用的同点行动骰；② 按普通主炮命中阈值计算的 2d6 命中率严格低于 50%。两项均满足时自动触发精确射击，额外消耗该同点骰，并令本次攻击最终命中所需点数 -2；命中率判断必须使用应用 -2 **之前**的普通阈值。AI 使用与玩家相同的瞄准表现：炮塔对准后停顿 0.5 秒，再播放开火音效与炮口火焰。迷雾内单位仍按完整炮塔转向时长推进并保留 0.5 秒停顿，只隐藏单位及转向画面；攻击方或被攻击方任意一方处于迷雾时跳过详细骰子界面。迷雾内攻击者开火时临时显形，并播放与可见单位相同的音效和炮口特效。若同点骰已使用、命中率达到或超过 50%、当前动作回退为机枪、行动单位不是坦克或处于经典模式，则不触发。

**转向规则（共 4 条优先级）**：
1. 当前目标在正前直线 + 前格可通行 → 不转向
2. 当前目标在正前直线 + 前格不可通行 → 转向到正前可通行格
3. 当前目标在正后直线 + 后格可通行 → 转向到正前可通行格
4. 否则朝"对准当前目标方向"角度最小的一边转向

AI 行动顺序：
- 友军队友坦克：每回合阶段⑥行动，按**距其当前目标最近 → 最远**排序；同距离随机。
- 德军坦克：每回合阶段⑦行动，按**距其当前目标最近 → 最远**排序；同距离随机。

- 各阶段骰子点数 → 动作规则与欧洲战场 §3.6.2 相同；只有 Pacific 章节前文明确列出的动作限制 / 覆盖规则生效，例如海滩格禁修复 / 烟雾、海滩格禁隐蔽、Pacific 机枪射击规则等。

**Pacific 杂项限制**

- 当玩家控制坦克当前处于海滩格时，不能执行修复与烟雾操作：
  - 杂项 4 点"修复"不可执行；
  - 杂项 5 点"烟雾 或 修复"中的两个选项都不可执行；
  - 其它来源若触发玩家坦克修复 / 烟雾，也应先检查该限制。
- 杂项阶段使用两个相同骰子进入隐蔽必须同时满足：
  - 玩家控制坦克**不在海滩格**；
  - 玩家控制坦克机动正常，即没有处于瘫痪等导致"不能移动"的状态。
- 若不满足上述条件，"进入隐蔽（+同点骰）"不出现在可选菜单中；若 UI 已显示该选项，也必须在确认时拒绝并提示原因，不消耗骰子。

**Pacific 美军伤亡检定**

- 太平洋战役每回合所有日军行动完毕后追加 1 个 Pacific 专属阶段：**美军伤亡检定**。它发生在常规阶段⑧"回合结束事件"之前；欧洲战场仍保持 §2.1 的 8 阶段流程。
- 关卡开始时 `usCasualties = 0`。该值代表战场整体美军伤亡压力，不等同于 Sherman 乘员死亡。
- 检定时遍历场上所有**存活日军单位**，按单位配置的 `usCasualtyDice` 数量投掷 d6：
  - Japanese Infantry 示例：`usCasualtyDice = 1`；
  - Type 97 Chi-Ha 示例：`usCasualtyDice = 2`；
  - AT Gun / Heavy Artillery 等单位的骰数单独配置，不写死在代码里。
- 每掷出 1 个 6 点，`usCasualties += 1`。
- UI 表现：进入美军伤亡检定阶段时弹出检定面板，按单位类型合并列出各存活日军单位提供的骰子数量（例如 `重炮*2：4颗`）、显示所有骰子点数，并对 6 点骰做高亮闪烁；结果行显示 `美军伤亡+X`，玩家点击确认后才进入后续回合结束事件或下一回合。
- 每个 Pacific 关卡可配置 `usCasualtyLimit`；当 `usCasualties > usCasualtyLimit` 时，当前关卡立即失败。A1 / B4 / D11 当前均配置为 10；未配置或配置为 0 时不显示该 HUD 行，也不触发此失败条件。
- 已摧毁 / 已移除的日军单位不再提供美军伤亡骰；被隐蔽、烟雾或朝向影响的单位只要仍存活，就按配置提供骰子。

#### 4.2.15 Pacific 行动表差异（待数据化）

- Pacific 可以拥有独立的玩家行动表：截图中出现 Clear / Tree / Beach 等地形列，不能复用欧洲 Road / Field / Mud 的全部骰池。
- Pacific 的日军 AI 不再使用欧洲 §3.7 的"起始格地形 → AI 列 / 掷骰数"规则；不同日军单位使用不同 AI 行动表，按单位类型 / profile 选择行动表。
- Pacific 日军行动骰数量也不再与当前地形或起始地形绑定，而是与单位类型 / profile 绑定：例如 Japanese Infantry、Type 97 Chi-Ha、AT Gun、Heavy Artillery 可分别配置自己的 AI 掷骰数。
- 日军步兵是可行动单位：在日军行动阶段按 Japanese Infantry 对应 AI 表和该单位类型的骰数投掷行动骰，执行移动 / 转向 / 射击等结果；它不再像欧洲战场德军步兵那样只通过回合结束事件行动。
- 日军行动表应支持"若不能射击则转向 / 移动 / 装填 / 空转"等备选链，且需要保留"接近 Sherman 时优先开火"的表内条件。
- 命中判定骰对子杀车长是通用规则，见 §3.4 Step 1；Pacific 日军单位攻击玩家 Sherman 时同样适用，但必须先命中成功，未命中时即便掷出对子也不会击杀车长。
- Pacific 美军伤亡检定的 `usCasualtyDice` 必须按单位类型配置到数据源中；同一单位在不同章节可拥有不同伤亡骰贡献。
- Pacific 的回合结束事件表可独立配置，例如狙击手、地雷、步兵增援、指挥官额外行动等；这些事件不回写欧洲章节。
- Pacific 支援点分两类：`eid`/`ef` 用于开局重炮 / 反坦克炮等初始炮位，`rid`/`rf` 用于回合结束事件刷新的步兵支援或坦克支援。欧洲战场的步兵援军也可配置 `rf`；未配置时保持旧兼容逻辑。
- 若某关的开局随机部署只允许某个敌方单位使用一部分 `eid`，在该敌人的 placement 上配置 `startEids` 白名单；未配置时沿用全局 `enemyDiceEidMax` / eid 1..6 链式占位。例：Pacific A2 中 Heavy Artillery 与 AT Gun 只在 `startEids: [5, 6]` 内随机，两个 Type 95 只在 `startEids: [1, 2, 3, 4]` 内随机。
- 若某关的开局随机部署要求单位使用蓝色编号 / `rid` 池，在该敌人的 placement 上配置 `startRids` 白名单；这不要求该单位是欧洲 `isFootKind`，因此可用于 Pacific A3 的 Japanese Infantry 初始部署。
- Pacific 日军坦克增援（Type 97）使用 `eid`/`ef` 支援点；若掷出的 `eid` 位于岩石高地（`terrain='rocky'`），该次增援无效，不链式改投其它编号。
- Pacific 海滩格（`terrain='beach'`）禁止使用烟雾、修复、隐蔽以及 Hull Down 类掩护；玩家菜单仍显示"烟/修"可选项，点击后按具体动作提示"在海里无法使用烟雾 / 修复 / 隐蔽"且不消耗骰子，敌方 AI 会把这些掩护动作视为不可执行并进入备选链。

- Japanese Infantry 的 `FIRE if adjacent to Sherman` 只在与当前敌方坦克目标相邻时射击；不相邻则本骰无效果。
- Japanese Infantry 的移动必须让它与当前目标距离减少，并按图中优先级选择候选格。当前实现将已存在的地图特性映射为：岩石高地 / 岩石格 > 带建筑格 (`bd:1`) > 树木 > 其它可入地形；海滩不可进入，同优先级随机。
- 移动目标：优先向当前目标移动 1 格；移动后的距离必须小于移动前距离，否则不移动。
- 阻挡与进入限制：日军步兵会被双方坦克阻挡，不能进入任意坦克占据的格子；日军步兵不可进入海滩格。
- 候选格筛选：只考虑无阻挡、可进入、且能让日军步兵与当前目标距离减少的相邻格。
- 候选格优先级：当多个候选格都能减少距离时，按 **岩石 / 岩石高地 > 带建筑格 (`bd:1`) > 树木 > 其它可入地形** 的优先级选择。
- 随机规则：若多个候选格的优先级相同，则从这些同优先级候选格中随机选择 1 格移动。

### 6.2 关键技术点

#### 6.2.1 六角格坐标
推荐采用 **Axial 坐标系**（q, r），渲染时转笛卡尔：
```ts
// pointy-top 六边形
const SIZE = 60;
function axialToWorld(q: number, r: number): Vec2 {
  const x = SIZE * Math.sqrt(3) * (q + r / 2);
  const y = SIZE * 1.5 * r;
  return new Vec2(x, y);
}
```
邻接 6 方向常量、距离公式（cube distance）封装在 `HexGrid.ts`。

#### 6.2.2 数据驱动任务
每关用 JSON 表示，包含：
- 地形矩阵（每格类型 + 是否树篱边）
- 初始单位放置（位置 + 朝向）
- 红色数字格（援军生成位）
- 行动表 / AI 表 / 事件表（覆盖默认值）
- 胜负条件（脚本表达式或枚举）
- 游戏模式不属于关卡数据；战争迷雾由主界面选择的 `GameMode` 决定（完整规则见 §3.3.1）
- 玩家车辆初始视野（`sherman.visionRange: number`，缺省 `4`；运行时可被天气等系统修改，存档保存当前值）

**复合目标 `destroy_kind_evac`**：在击毁 `objective.kind` 所指全部敌方单位后，谢尔曼须从 **`evacAt`** 六角格沿 **`evacExitDir`** 六向之一执行 **前进** 或 **后退**（后退的位移方向 = `facing+3`，须等于 `evacExitDir`），且 **邻格不在地图内**（无 `Tile`）时撤离成功并判胜；动画结束后谢尔曼坐标可落在网格外。存档字段 `shermanEvacuated` 由 `SaveLoad` 持久化。若配置 **`objective.kinds`**（`UnitKind[]`，非空），则须将所列 **每一种** kind 的敌方单位**全部**击毁后才允许撤离（与单独的 `kind` 二选一：有 `kinds` 时优先用 `kinds`，典型见任务 9：虎式 + IV 号，步兵不计入）。

**徒步类单位（步兵 / 军官，`isFootKind` / `isFootUnit`）**：`UnitKind` 中 `'infantry'`（步兵）与 `'officer'`（军官，任务 8 起）共享「徒步类」语义——`size = 0`，无装甲，仅可被机枪打死、不参与坦克 AI、不渲染装甲 / 装填面板。占格规则按阵营判定：坦克 / 卡车可与**己方**徒步单位叠格，但不能进入**敌对**徒步单位所在格；例如玩家谢尔曼不能驶入德军步兵格，德军坦克也不能驶入玩家阵营步兵格。`types.ts` 导出的 `isFootKind(kind)` / `isFootUnit(unit)` 是这两类共有判定的统一入口；`Combat.canMGAttack`、`TurnEndEventApply.adjacent_infantry_fire`、`BattleScene` 的目标筛选 / 渲染分支等都通过它们进行兼容判定。**唯一例外**：`infantry_spawn` 事件 spawn 出来的单位 `kind` 始终为 `'infantry'`（不会复活军官），所以「按 `kind` 精确判定」的位置（如 `Objective.allEnemiesOfKindDestroyed(mission, 'officer')`）不能换成 helper。

**Pacific 日本步兵（`japanese_infantry`）**：不是欧洲「徒步类」单位，仍会按日军 AI 表行动，并拥有 Pacific 专属体型 / 装甲 / 穿甲；但玩家机枪扫射把它视为步兵目标，不限射程，按 Pacific 机枪 1d6 命中流程结算。

**新单位 `'officer'`（军官）**：与 `'infantry'` 数值一致（见 `data/units.csv`），但 `kind` 独立——典型用法见任务 8（`mission_08.json`）：建筑里固定放置 1 只军官，objective 为 `{ type:"destroy_kind_evac", kind:"officer", evacAt, evacExitDir }`，与回合结束 5–6 spawn 出来的普通步兵互不混淆。UI 会在军官身周加红色光环、并在所在格绘制红色 hex 边框（`OFFICER_HALO_STROKE` / `OFFICER_TILE_STROKE`），与说明书原图「红色边框建筑里的德军步兵」一致。

**敌方掷骰出生（可选，`enemyStartByDice: true`）**

适用于「开局时敌方位置不完全固定」的关卡。地图上用 **`eid`**（1～6，全图**不重复**）标记**坦克 / 炮位等初始出生位**；用 **`rid`**（1～6，全图**不重复**）标记**回合结束支援 / 援军出生位**。`eid` 的朝向用 **`ef`**（0=E … 5=NE）配置；`rid` 的朝向用 **`rf`** 配置。方向编号与**同格** `h` 的 6 位、以及 `HEX_DIRECTIONS[i]` **完全一致**（自本格心指向第 `i` 邻格心 = 经该边越境的六向）。树篱位 `h[i]=1` 与 `ef` / `rf` 均用**轴向**同一套索引 i，与 `HexGrid.hedgeFlagsFromMapJson` 及 `HEX_DIRECTIONS` 一致；Pacific 防波堤 **`bw`** 同样为 6 位轴向 `0/1`，规则层按同一物理边双侧识别。`BattleScene` 绘制时经 `HEDGE_DRAW_EDGE_BY_AXIAL` 将轴向 i 映到 `drawHedgeEdge` / 防波堤边的**几何边**号。**公路 / 机场视觉**：可选 **`rd`**（6 位 `0/1`，**仅** `t='r'`、`t='a'` 或水域+桥梁可配）按同一轴向索引绘制方向条带；公路格绘制"边中点 → 格心"的道路条带，当只有 1 位为 `1` 时，格心额外绘制圆形"道路尽头"图案；机场格读取 `rd` 的直线轴向，绘制更宽且两端各缩进约 1/5 格的跑道。`rd` **不影响**移动 / 视线 / 骰子规则，纯视觉字段（解析见 `HexGrid.roadFlagsFromMapJson`，渲染见 `BattleScene.drawRoadOverlay` / `drawAirstripOverlay`）。Pacific 深海使用 `t='dw'`，机场使用 `t='a'`。

- 关卡 JSON 中 **`enemyStartByDice: true`** 时，每个 **`enemies[]`** 条目可省略 `at` / `facing`，由运行时掷骰决定。
- **谢尔曼掷骰（可选，`shermanStartByDice: true`）**：与坦克共用地图上的 **eid 1..6** 黑格表，须同时 **`enemyStartByDice: true`**；`sherman` 可省略 **`at` / `facing`**，由 `MissionLoader` 在加载时**先于**德军无坐标单位掷 1d6 链式择空位（与坦克规则相同，占位后写入格上 `ef` 朝向）。典型任务见 `mission_11.json`（钢铁角斗士）。
- **`kind: infantry`**：**每名无 `at` 的步兵开局掷 1 颗 d6**，链式占用 **`rid = 骰点` → `rid+1` … 绕回 1** 的六角格（规则与下款相同，但查 **`rid` 表**，不是 `eid`）；地图上须配置至少一处 **`rid`**，否则加载失败。若该 `rid` 配有 `rf`，步兵朝向使用 `rf`；旧图未配时兼容回退到同格 `ef` / 0。
- **坦克等非步兵**：**每名无 `at` 的单位开局掷 1 颗 d6**，首先尝试落在 **`eid` 等于骰点** 的六角格；若该格已被占用（含已放置的其它敌方单位，或谢尔曼初始格），则改试 **`eid = 骰点 + 1`**（6 的下一档为 1），再重复直至找到**既存在 `eid` 配置、又未被占用**的格；若连续 6 档仍无法落位则加载失败（配置错误）。
- 未配置对应编号的格在链式探测中跳过（例如关卡只配置了 `eid` 1、2，掷出 5 时会依次尝试 5→6→1→2…直到命中已配置且空闲的格；**`rid` 链同理**）。
- 实现见 `MissionLoader.loadMission(data, rng)`：需在创建任务后传入与整局共用的 **`RNG`**（如战斗场景种子），以便与后续随机流一并复现。

### 6.3 核心规则伪代码示例

```ts
// Combat.ts
export interface AttackParams {
  attacker: Unit;
  target: Unit;
  distance: number;
  hedgeCount: number;
  inBuilding: boolean;
  inSmoke: boolean;
  isHidden: boolean;
  fromBack: boolean;
  commanderHatchOpen: boolean;
}

export function resolveAttack(p: AttackParams, rng: RNG): AttackResult {
  // Step 1: hit
  const need = p.target.size + p.distance + p.hedgeCount
             + (p.inBuilding ? 1 : 0)
             + (p.inSmoke ? 1 : 0)
             + (p.isHidden ? 2 : 0)
             + (p.fromBack ? 1 : 0);
  const d1 = rng.d6(), d2 = rng.d6();
  const hitSum = d1 + d2;
  const isHit = hitSum > need;
  if (!isHit) return { hit: false };

  // Doubles + open hatch → commander dies
  const isDoubles = (d1 === d2);

  // Step 2: damage?
  const armor = pickArmor(p.target, p.attacker.facingRelative);
  const damageRoll = rng.d6();
  const causeDamage = damageRoll >= armor - p.attacker.penetration;
  if (!causeDamage) return { hit: true, doubles: isDoubles, damaged: false };

  // Step 3: damage type
  const typeRoll = rng.d6();
  const damageType = lookupDamageTable(p.target === p.protagonist, typeRoll);
  return { hit: true, doubles: isDoubles, damaged: true, damageType };
}
```

### 6.5 多语言与文案管理（lang.csv）

**硬规则**：**游戏里所有用户能看到的文本**（HUD、按钮、弹窗、浮字、状态面板、骰子托盘提示、伤害 / 乘员结算、胜负标语、攻击失败原因等）**都必须通过 `lang.csv` 读取，不允许在代码里硬编码中文 / 英文字符串。**同时游戏需支持运行时切换语言，默认显示中文。

#### 6.5.1 工程布局

| 文件 | 角色 |
|------|------|
| `data/lang.csv` | **文案主表**（策划直接维护）。UTF-8 + BOM，3 列：`key,zh,en` |
| `tools/buildLangDB.js` | CSV → TS 生成器。零依赖 Node 脚本 |
| `assets/scripts/core/LangDB.ts` | **自动生成**产物，不可手改（会被覆盖） |
| `assets/scripts/core/Lang.ts` | 运行时入口：`t()` / `setLang()` / `getLang()` |

#### 6.5.2 CSV 约定
- 3 列固定顺序：`key,zh,en`。三列都不允许为空，否则生成脚本拒绝通过。
- `key` 全局唯一，建议用点号分组命名：`btn.nextPhase`、`hud.movePhase`、`dmg.effect.fire`、`crew.role.1`。
- 值里允许 `{name}` 形式的占位符，调用 `t(key, { name: value })` 替换。
- 值里允许字面量 `\n`（反斜杠 + n），运行时自动展开为真正的换行，方便 Excel 编辑多行文案。
- 如果值里需要逗号或换行，按标准 CSV 用双引号包裹字段（`""` 转义内部引号）。

#### 6.5.3 运行时 API
```ts
import { t, setLang, getLang } from '../core/Lang';

label.string = t('btn.nextPhase');                             // 下一阶段
label.string = t('hud.movePhase', { n: 3, dice: '4/5' });      // 回合 3 | 移动阶段 | 骰子 4/5
label.string = t('preview.impossible', { n: 12 });             // 两行文案，\n 自动换行

setLang('en');   // 一行切到英文，下一次 t() 调用立即返回英文
```
- 找不到 key 时 `console.warn` 并原样返回 key，线上也能一眼看出缺漏。
- 默认 `currentLang = 'zh'`，语言切换可以绑定到主菜单的"设置"入口。

#### 6.5.4 策划工作流
1. 用 Excel 打开 `data/lang.csv` → 编辑 `zh` / `en` 两列（`key` 列别动）。
2. "另存为 CSV UTF-8（逗号分隔）" 覆盖原文件。
3. 在项目根目录执行 `node tools/buildLangDB.js`，自动重写 `LangDB.ts`。
4. 在 Cocos Creator 里预览即可看到新文案；无需重启编辑器。

#### 6.5.5 写代码的硬约束
- **绝不在 `.ts` 里硬编码中/英文字符串给任何 `label.string` / `button.text` / `spawnFloater()` 文案参数**。新增文案 → 先进 `lang.csv` → 跑生成器 → 再在代码里 `t('your.key')`。
- `core/` 层函数如果需要返回"原因类文本"（如 `canAttack().reason`），**只返回 i18n key**（参见 `Combat.AttackDenyReason`），由 UI 层统一 `t(reason)` 渲染。核心层与文案必须零耦合。
- 控制台日志 (`console.log` 等) 允许保留中英文硬编码——仅供开发者调试，不视为玩家可见文本。
- 编辑器 Inspector 的 `@property({ tooltip })` 同理：只是策划工具，不算玩家可见文本。

### 6.6 数值数据管理（CSV 驱动）

**硬规则**：**游戏里所有可调数值**（单位装甲 / 穿甲 / 体型、敌方 AI 行动表、伤害表、地形移动成本、骰子池大小、任务奖励、乘员加成……）**都必须配置到 `data/*.csv` 里，由生成脚本编译成 `assets/scripts/core/*DB.ts`，禁止把数字魔法常量散在代码中。**

目的：策划打开 Excel 改格子就能调参，不用等程序员改代码。

#### 6.6.1 现有 / 规划中的 CSV 表

| CSV | 生成器 | 输出 | 内容 |
|-----|--------|------|------|
| `data/units.csv` | `tools/buildUnitDB.js` | `core/UnitDB.ts` | 单位类型：阵营 / 体型 / 4 个装甲面 / 穿甲 / 美军伤亡骰 / 移动音效 / 攻击音效 |
| `data/lang.csv` | `tools/buildLangDB.js` | `core/LangDB.ts` | 所有玩家可见文本（见 6.5）|
| `data/player_action_table.csv` + `data/player_dice_pool.csv` | `tools/buildPlayerActionDB.js` | `core/PlayerActionDB.ts` | §3.6 玩家行动表（§3.6.2：1..6 + 对子 → A/B/C）+ §3.6.1 掷骰数（子阶段 × 地形基础 + 按阶段乘员 / 舱盖修正 + 上下限）|
| `data/enemy_ai_table.csv` + `data/enemy_ai_dice.csv` | `tools/buildEnemyAIDB.js` | `core/EnemyAIDB.ts` | §3.7 德军坦克 AI 表（列 × 骰面 → 主动作 / 备选动作）+ 每列骰数（road 4 / field 4 / mud 3 / damaged 2）；Pacific 追加按单位类型的 AI 列（type97 / at_gun / japanese_infantry / heavy_artillery）|
| `data/terrain.csv`（规划）| `tools/buildTerrainDB.js` | `core/TerrainDB.ts` | 地形移动成本 / LoS 阻挡 / 射击修正 |
| `data/damage_table.csv`（规划）| `tools/buildDamageDB.js` | `core/DamageDB.ts` | 1d6 伤害点数 → 效果（起火/炮塔/痛痪/阵亡检定/击毁）|

> **章节单位数值**：当同一单位在不同战场使用不同规则体系时，单位数据必须支持章节 / 战场维度。典型例子是 M4 Sherman：欧洲战场 Step 2 使用 1d6 击穿，因此装甲值较低；太平洋战场 Step 2 使用 2d6 击穿，因此 Sherman 与日军单位都需要 Pacific 专属装甲值。实现可通过 `chapter` / `theater` 字段、独立 kind，或拆分 `data/units_europe.csv` / `data/units_pacific.csv` 达成，但生成后的 `UnitDB` 必须能按任务所属战场选择正确 profile。

> **Pacific Sherman 数值**：太平洋战役的玩家 M4 Sherman 使用独立 profile：体型 4、前装甲 11、前侧 10、后侧 9、后装甲 8、穿甲 2。该 profile 不覆盖欧洲 Sherman。

> **Pacific 步兵数值**：Japanese Infantry 不能套用欧洲徒步类单位的 `size=0 / 无装甲 / 只由机枪打死` 语义；它需要像坦克一样配置体型、四向装甲和穿甲值（当前规格：体型 4、四向装甲 0、穿甲 3），并参与 Pacific 的命中 / 击穿流程。

> **Pacific 玩家骰池**：太平洋战役的玩家 Sherman 基础骰数需要在 `player_dice_pool` 或等价章节化数据源中单独配置；行动骰面规则沿用欧洲 §3.6.2，乘员加成沿用欧洲 §3.6.1，但基础骰数按 Pacific 基底地形读取（沙滩 / 树木 / 海滩 / 岩石高地）。带建筑格不是独立地形，配置方式与欧洲战役相同：在基底格上叠加 `bd:1`，并继续提供建筑命中修正与中间格阻挡视线等格子特性。

> **Pacific 日军 AI 表与骰数**：太平洋战役的日军 AI 行动表和 AI 掷骰数必须支持按单位类型 / profile 查询。欧洲战场可继续使用 `road / field / mud / damaged` 等地形列；Pacific 则由 `unitKind` 或等价字段选择行动表和骰数，不从地形推导。

> **Japanese Infantry AI 配置**：Japanese Infantry 的 AI 骰子数固定由单位 profile 配置为 3；其 AI 表需要表达骰面 1 / 3 / 6 攻击相邻敌军、骰面 2 / 4 / 5 移动 1 格，并支持移动候选格的地形优先级（岩石 / 岩石高地 > 带建筑格 `bd:1` > 树木 > 其它可入地形）与同优先级随机。

> **Pacific 美军伤亡骰**：太平洋战役的每种日军单位还需要配置 `usCasualtyDice`（或等价字段），用于回合最后的美军伤亡检定。该字段同样按章节 / 战场维度读取，例如步兵可配置为 1，坦克可配置为 2，固定炮和重炮单独配置；欧洲单位可留空或为 0。

> **单位阵营与音效配置**：每种 `unitKind` 的默认阵营写在 `data/units.csv` 的 `faction` 字段中（`allied` / `german` / `japanese`）。关卡 JSON 仍可显式填写 `faction` 以兼容旧数据；未填写时由 `UnitDB` 补齐。单位移动和攻击音效分别由 `moveSound` / `attackSound` 配置，路径为 `resources` 下无扩展名资源路径；字段为空字符串时不播放对应音效。
| `data/missions/*.json` | *(不走 CSV)* | *(直接 JSON)* | 单关地图 + 初始摆放 + 关卡级阈值（如 Pacific `usCasualtyLimit`）；结构太复杂，继续用 JSON |

> 命名约定：CSV 文件统一放 `data/` 下，生成器统一放 `tools/build*DB.js`，输出统一到 `assets/scripts/core/*DB.ts` 并带"自动生成，请勿手改"注释头。

#### 6.6.2 生成器通用约定
1. **CSV 自动规范化**：所有 `data/*.csv` 生成器必须通过 `tools/csvSmart.js` 读取源表；生成器需兼容 UTF-8 / UTF-8+BOM / GBK，以及逗号 CSV / Tab 分隔 TSV。每次成功读取后，源文件必须自动回写为 **UTF-8+BOM + 逗号 CSV**，保证 git diff 稳定、Excel 双击打开不乱码也不挤成一列。
2. **强校验**：
   - 关键列（枚举类 key / 外键引用）必须落在预设集合里，否则生成器报错并**拒绝写文件**，不允许污染游戏代码。
   - 数值列必须是合法整数 / 浮点数，空值或非数字会被精确定位到 CSV 行号。
   - 唯一约束：key / 主键列不允许重复。
3. **零依赖**：只用 Node 18+ 内置 `fs` / `TextDecoder`，不引任何 npm 包；CI 环境和本地都能直接跑。
4. **稳定 diff**：生成的 TS 输出列顺序 / 对齐 / 注释格式固定，让 git diff 只反映真实的数值变动。

#### 6.6.3 代码侧读取约定
- 运行时所有数值查询**只走生成出来的 `*DB.ts` 导出函数**（如 `getUnitStats(kind)`、`t(key)`），不允许 `import` 原始 CSV。
- 不允许在 `.ts` 里写如 `const SHERMAN_ARMOR = 6;` 这种魔法数字；改成 `getUnitStats('sherman').armorFront`。
- 如果某个数值还没上 CSV（比如仍在玩法探索期），代码里写 `TODO(data-csv): 上表名.csv` 注释，列入下一次数据化任务。

#### 6.6.4 策划工作流
```
Excel 开 data/XX.csv → 改数值 → 另存为 CSV UTF-8 →
  node tools/buildXXDB.js → Cocos 预览即时生效
```

脚本失败时终端会打印具体到 CSV 行号的原因（`第 5 行 unitKind="..."：字段 armorFront="abc" 不是非负整数`），定位 1 秒搞定。
